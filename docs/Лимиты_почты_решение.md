## Как это делают без кода (концепт)

### Вариант A: Failover (рекомендую)
- Основная почта: `mailbox_1`.
- При временной ошибке `450/4.2.1` на `mailbox_1`:
  - эту конкретную отправку пробуем через `mailbox_2`.
- Через некоторое время снова возвращаемся к `mailbox_1`.

Плюсы:
- простая логика;
- меньше потерянных отправок.

Риск: **средний** (нужно аккуратно считать попытки, чтобы не задвоить письма).

---

### Вариант B: Round-robin / распределение нагрузки
- Отправки сразу делятся между 2 ящиками (например 50/50).
- Это снижает шанс упереться в лимит одного получателя/ящика.

Плюсы:
- меньше пиков;
- равномерная нагрузка.

Риск: **средний** (сложнее мониторить, кто что отправил).

---

### Вариант C: Circuit breaker
- Если подряд N ошибок `450` на ящике №1:
  - “отключаем” его на cooldown (например 10-30 минут),
  - отправляем только через ящик №2.
- После cooldown делаем пробную отправку через №1.

Плюсы:
- очень устойчивая схема в проде.

Риск: **средний/высокий** по сложности реализации, но лучший по надежности.

---

## Что важно учесть обязательно

- Оба ящика должны быть “доверенными” (SPF/DKIM/DMARC, app-password).
- В логах нужно хранить, **какой ящик** отправлял письмо.
- Нужна идемпотентность (чтобы один лид не ушел дважды).
- Лимиты могут быть не только у отправителя, но и у **получателя** (`recipient rate limit`) — тогда иногда помогает только замедление + пауза.

Да, в твоем примере можно понять довольно уверенно.

Текст ошибки:
`450 4.2.1 The recipient has exceeded message rate limit`

Это указывает на лимит **получателя** (`recipient`), то есть ограничения на стороне адреса, куда ты отправляешь (в твоем случае `npp-rekord@yandex.ru`), а не отправителя.

Как ориентироваться в целом:

- `recipient has exceeded...` / `too many messages to recipient`  
  -> лимит **получателя**.
- `sender has exceeded...` / `authentication` / `quota for account`  
  -> лимит/проблема **отправителя**.
- `4xx` (например `450`)  
  -> обычно временная ошибка, стоит ретраить.
- `5xx`  
  -> чаще постоянная/жесткая проблема (но не всегда).

Важно: SMTP-тексты зависят от провайдера, но у тебя формулировка прям явная — **recipient rate limit**.

Коротко: **обычно лимит “recipient rate limit” привязан к получателю**, то есть к ящику `npp-rekord@yandex.ru`, а не только к одному отправителю.

Что это значит на практике:

- Отправка с другой твоей почты на **тот же** `npp-rekord@yandex.ru` может **не помочь** (или помочь слабо), если уперся именно лимит входящего потока у получателя.
- Иногда у провайдера лимит комбинированный (получатель + отправитель + временное окно), но текст `recipient has exceeded...` чаще про сторону получателя.

Как действовать в таком кейсе:
- замедлять отправку (`EMAIL_SEND_DELAY_SECONDS`),
- увеличивать ретраи/растягивать их по времени,
- по возможности отправлять на альтернативный получатель (если бизнес допускает),
- не “долбить” частыми повторами в коротком окне.

Итого: второй отправитель на тот же адрес — **не гарантированное решение** для твоей конкретной ошибки.

Оценка риска:
- **Средний**: письма не теряются сразу, но часть может дойти до `failed`, если окно лимита долгое.
